# court-guard（日本語版）

English: [README.md](README.md)

Claude Code / Opus 4.8 の**ツール呼び出し破損バグ**（通称 court/invoke 漏れ）を対象に、発生の緩和、漏れの検知、漏れた後の復旧を行うプラグイン。
このバグでは、ツール呼び出しの開始タグが生成時に壊れ、呼び出しは実行されないまま、ツール呼び出しのXMLが返答の地の文に漏れる。
一度漏れると会話履歴が汚染され、モデルは漏れたテキストを真似て同じ壊れ方を繰り返す。
報告が多いのは Windows 上の Opus 4.8（1Mコンテキスト）である。

## できることとできないこと

バグはモデル自身のトークン生成（デコード）の中で起きる。
Claude Code のプラグインはハーネス側で動くため、壊れたトークンが生成されている最中に介入できる層が存在しない。
したがって court-guard は破損そのものを防げない。
できるのは、引き金を引く頻度を下げることと、漏れた後の被害を短く抑えることである。

| できること | できないこと |
|---|---|
| ターン終端で漏れXMLを検知し（本線とサブエージェントの両方）、1回だけクリーンな撃ち直しを強制する | 壊れたトークン生成そのものの防止 |
| 履歴が汚染されたら、毎プロンプト「真似するな、引数を短く」を注入し続ける | 生成ストリームの途中での介入 |
| 長いツール引数（＝引き金）から離れるよう、実行を止めずに助言を注入する | 履歴汚染の除去（漏れたテキストは残る） |
| Write/Edit がディスクに届いていない「幻の書き込み」を警告する（実験的） | Read/Grep の幻出力の検知（比較対象がない） |

## 中身（v0.3.x）

- **leak_detector**（Stop + SubagentStop）：最終アシスタントメッセージを走査し、漏れたツール呼び出しXMLを検知する。既定は `retry` モードで、ターンを1回だけブロックしてモデルにクリーンな撃ち直しをさせる。ブロックの `reason` はモデルに実際に届く唯一の経路である（`systemMessage` の警告はユーザーにしか見えない。旧既定の `warn` が自動復旧に無力だった理由がこれである）。SubagentStop では `last_assistant_message` の提供が公式に明記されていないため、transcript 末尾を読む予備経路で対応し、サブエージェント内の漏れも捕捉して撃ち直させる。コードフェンスとインラインコード内の引用は誤検知として除外する。`stop_hook_active` によりループしない。
- **contamination_notice**（UserPromptSubmit）：セッションの transcript に漏れXMLが既にある場合、毎プロンプト、モデルの文脈に注意を注入し続ける。内容は、壊れたテキストを真似るな、全ツール引数を短く、再発したら再起動を勧めろ、の三点。1回の漏れが何度も再生産される「模倣ループ」への対抗策である。
- **heavy_cmd_guard**（PreToolUse）：長いツール引数を監視する。対象はシェルコマンドに加え、長い `Write` content、`Edit` new_string、`Agent` prompt。既定は助言モードで、deny しない。このフックに届いた呼び出しはパースに成功している、つまり漏れなかった呼び出しであり、それを弾くと同じ内容の再生成（長い生成がもう1回）を強いるからである。代わりに実行は通し、`additionalContext` で以後の呼び出しを短くするよう誘導する。`COURT_GUARD_ENFORCE=1` の強制モードでは「削れる長さ」に限って deny する（対象と理屈は「設定」を参照）。履歴が既に汚染されている場合は助言を格上げする。残りの塊は短い参照型 prompt でサブエージェントへ渡す（サブの文脈はクリーンな状態から始まる）、ファイル更新は地の文で出してユーザーに手貼りしてもらう、それも無理なら再起動を勧める。「サブエージェントを使え」が機械的に正解になる唯一の局面である。
- **write_verify**（PostToolUse、実験的）：Write/Edit が成功と報告されたのに、対象ファイルが存在しないか空のとき警告する。
- **/court-guard:court-recover**（コマンド）：手動復旧の手順書。1回だけ撃ち直す、/compact しない、直らなければ再起動する、過去の書き込みは現物を確認する。

## インストール

```
/plugin marketplace add sakuranjunkie-staff/court-guard
/plugin install court-guard@court-guard
```

## 設定（環境変数）

- `COURT_GUARD_ENFORCE` = `1`：heavy_cmd_guard が「削れる長さ」だけを拒否（deny）する。対象はシェルコマンド（一呼び出し一仕事に割っても内容は損なわれない）と Agent prompt（内容は常にファイル参照で渡せる）。長い Write/Edit の中身は、履歴が汚染されていない平時は弾かない。新しい文書やコードは削れない長さであり、どこかのツール呼び出しを通るしかなく、弾いても同じトークンが分割 Edit に化けて失敗要因が増えるだけだからである。Write/Edit を弾くのは履歴が汚染済みのときだけで（そこでの長い書き直しは再漏れすると実測済み）、指示は手貼りかサブエージェント委譲になる。deny を使う根拠：助言は従われないことがあるが、deny は従うまで進めない。しかも従った結果の短い呼び出しが履歴に並び、モデルが真似る手本が良い方に置き換わる。既定は無効（助言モード）。
- `COURT_GUARD_MODE` = `retry`（既定）または `warn`：自動で1回撃ち直すか、ユーザー向けの警告だけを出すか。warn はモデルに届かない。
- `COURT_GUARD_CMD_MAXLEN` = 整数（既定 `200`）：シェルコマンド長の助言閾値。
- `COURT_GUARD_ARG_MAXLEN` = 整数（既定 `3000`）：Write/Edit/Agent 引数長の助言閾値。
- `COURT_GUARD_SCRIPTS_HINT` = 自由テキスト（任意）：deny/advisory の分割ヒント末尾に追記される。ローカルのスクリプト索引（例「先に scripts/README.md を見ろ」）を指しておくと、長い本文の書き直しでなく既存スクリプトの再利用へ誘導できる。
- `leak_patterns.txt`（プラグインデータディレクトリ `$CLAUDE_PLUGIN_DATA` に置く）：破損が自分のログでどう見えるかに合わせて追加する正規表現。1行1パターン。

## 注意事項

- **誤検知**：返答がコードフェンスの外でツール呼び出しXMLを正当に論じている場合、`leak_detector` と `contamination_notice` が発火しうる（バッククォートやフェンス内の引用は無視される）。`retry` モードでの最悪ケースは余計なターンが1回入るだけで、ループはしない。
- `retry` は1回だけ撃ち直す。撃ち直しも漏れたら court-guard は手を引き、再起動を勧める。
- 最大のレバーはプラグインの外にある。このバグの報告は Opus 4.8 の 1M コンテキストティアに集中している。標準の 200k ティア（`/model claude-opus-4-8`、`[1m]` なし）で作業できるならそうする。汚染された履歴への唯一の根治は再起動であり、セッションは惜しまず切り直す。
- `python` が PATH に必要。Windows では、Claude Code が使うシェルで `python` が解決することを確認する。
