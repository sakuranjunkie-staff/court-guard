# court-guard（日本語版）

Claude Code / Opus 4.8 の**ツール呼び出し破損バグ**（通称 court/invoke 漏れ）に対する緩和・検知・復旧プラグイン。ツール呼び出しの開始タグが生成時に壊れ、呼び出しは実行されず、ツール呼び出しのXMLが返答の地の文に漏れる。一度漏れると会話履歴が汚染され、モデルはそれを真似て同じ壊れ方を繰り返す。報告が多いのは Windows 上の Opus 4.8（1Mコンテキスト）。

## このプラグインはバグを「治せない」

バグはモデル自身のトークン生成（デコード）の中で起きる。Claude Code のプラグインはハーネス側で動くため、壊れたトークンが生成されている最中に介入できる層が存在しない。**court-guard は破損そのものを防げない。** できるのは、引き金を引く頻度を下げることと、漏れた後の被害時間を最短にすることだけだ。

| できること | できないこと |
|---|---|
| ターン終端で漏れXMLを検知し（本線・サブエージェント両方）、1回だけクリーンな撃ち直しを強制する | 壊れたトークン生成そのものの防止 |
| 履歴が汚染されたら、毎プロンプト「真似するな・引数を短く」を注入し続ける | 生成ストリームの途中での介入 |
| 長いツール引数（＝引き金）から離れるよう、実行を止めずに助言を注入する | 履歴汚染の除去（漏れたテキストは残る） |
| Write/Edit がディスクに届いていない「幻の書き込み」を警告する（実験的） | Read/Grep の幻出力の検知（比較対象がない） |

## 中身（v0.2.x）

- **leak_detector**（Stop + SubagentStop）：最終アシスタントメッセージを走査し、漏れたツール呼び出しXMLを検知する。デフォルトは `retry` モード＝ターンを1回だけブロックし、モデルにクリーンな撃ち直しをさせる。ブロックの `reason` は**モデルに実際に届く唯一の経路**だ（`systemMessage` の警告はユーザー表示のみで、モデルには見えない——旧デフォルトの `warn` が自動復旧に無力だった理由）。SubagentStop では `last_assistant_message` の提供が公式に明記されていないため、transcript 末尾を読む予備経路で対応——サブエージェント内の漏れも捕捉して撃ち直させる。コードフェンス・インラインコード内の引用は誤検知除外。`stop_hook_active` によりループしない。
- **contamination_notice**（UserPromptSubmit）：セッションの transcript に漏れXMLが既にある場合、毎プロンプト、モデルの文脈に注意を注入し続ける——壊れたテキストを真似るな・全ツール引数を短く・再発したら再起動を勧めろ。1回の漏れが何度も再生産される「模倣ループ」への直接の対抗策。
- **heavy_cmd_guard**（PreToolUse・v0.2で全面再設計）：**もう deny しない**。このフックに届いた呼び出しはパースに成功している＝漏れなかった呼び出しであり、それを弾くと同じ内容の再生成（＝長い生成がもう1回）を強いてしまう。代わりに実行は通し、`additionalContext` で「以後の呼び出しを短くしろ」と誘導する。監視対象はシェルコマンドだけでなく、長い `Write` content・`Edit` new_string・`Agent` prompt も含む。**履歴が既に汚染されている場合は助言を格上げ**——残りの塊は短い参照型 prompt（「ファイルXを読んでYをやれ」）でサブエージェントへ（サブの文脈はクリーンな状態から始まる）、ファイル更新は地の文で出してユーザーに手貼りしてもらう、それも無理なら再起動を勧める。「サブエージェントを使え」が機械的に正解と言い切れる唯一の局面。
- **write_verify**（PostToolUse・実験的）：Write/Edit が成功と報告されたのに対象ファイルが存在しない・空のとき警告する。
- **/court-guard:court-recover**（コマンド）：手動復旧の手順書（1回だけ撃ち直す・/compact するな・直らなければ再起動・過去の書き込みを現物確認）。

## インストール

```
/plugin marketplace add sakuranjunkie-staff/court-guard
/plugin install court-guard@court-guard
```

## 設定（環境変数）

- `COURT_GUARD_MODE` = `retry`（デフォルト）または `warn` ——自動1回撃ち直し vs ユーザー向け警告のみ（warn はモデルに届かない）。
- `COURT_GUARD_CMD_MAXLEN` = 整数（デフォルト `200`）——シェルコマンド長の助言閾値。
- `COURT_GUARD_ARG_MAXLEN` = 整数（デフォルト `3000`）——Write/Edit/Agent 引数長の助言閾値。
- プラグインデータディレクトリ（`$CLAUDE_PLUGIN_DATA`）の `leak_patterns.txt` ——自分のログでの破損の見え方に合わせた追加正規表現（1行1パターン）。

## 注意事項

- **誤検知**：返答がコードフェンスの**外**でツール呼び出しXMLを正当に論じている場合、`leak_detector`/`contamination_notice` が発火しうる（バッククォートやフェンス内の引用は無視される）。`retry` モードでの最悪ケースは余計なターンが1回入るだけで、ループはしない。
- `retry` は1回だけ撃ち直す。撃ち直しも漏れたら court-guard は手を引き、再起動を勧める。
- **最大のレバーはプラグインの外にある**：このバグの報告は **Opus 4.8 の 1Mコンテキストティア**に集中している。標準の200kティア（`/model claude-opus-4-8`・`[1m]` なし）で作業できるならそうして、セッションは気軽に再起動しろ——再起動が汚染された履歴への唯一の根治だ。
- `python` が PATH に必要。Windows は Claude Code が使うシェルで `python` が解決することを確認。
